# Skill Template - 通用 Skill 模板

> 新建 Skill 时使用此模板  
> **版本**: 1.0.0  
> **日期**: 2026-04-17

---

## 什么是完整 Skill

论文定义：完整 Skill = 操作过程 + 决策启发 + 规范约束

```
Skill = Operation Process + Decision Heuristics + Normative Constraints

操作过程     → 做什么（步骤骨架）
决策启发     → 怎么做（分支判断）
规范约束     → 做到什么程度算合格
```

**三者缺一不可**：
- 只有操作过程 → 会做但不会判断
- 只有决策启发 → 有判断依据但不知道从哪步开始
- 只有规范约束 → 知道标准但不知道怎么做

## 使用方法

```bash
# 复制模板到新 Skill 文件
cp core/harness/base/tasks/skill-template.md core/harness/base/tasks/[新skill名].md

# 然后按三节结构填充
```

## 第一节：操作过程 (Operation Process)

```markdown
## 第一节：操作过程

### CP0: 初始化
<!-- 这个 Skill 在什么情况下被触发，初始输入是什么 -->

### CP1: 规划
<!-- 规划阶段要做什么，输出什么 -->

### CP2: 执行
<!-- 具体执行步骤，按顺序列出 -->

### CP3: 验证
<!-- 如何验证执行结果 -->

### CP4: 完成
<!-- 完成时做什么收尾工作 -->
```

**注意**：操作过程只描述"做什么"，不描述"怎么做"。  
"怎么做"是第二节决策启发的事。

---

## 第二节：决策启发 (Decision Heuristics)

```markdown
## 第二节：决策启发

### 情况 A
```
触发条件 →
  ├─ 条件 1 → 动作 A
  ├─ 条件 2 → 动作 B
  └─ 条件 3 → 升级人工介入
```

### 情况 B
<!-- 更多分支情况 -->
```

**决策树格式**：
```
条件 → 动作
```
- 一个条件对应一个动作
- 多个条件用 `├─` 和 `└─` 区分优先级
- 无法判断时走最底层：人工介入

**常见决策场景**：
| 场景 | 触发条件 |
|------|---------|
| 阻塞 | 30 分钟无进展 |
| 范围蔓延 | 发现需求外的问题 |
| 方案分歧 | 多个可行方案 |
| 验证失败 | CP3 Gate 不通过 |

---

## 第三节：规范约束 (Normative Constraints)

```markdown
## 第三节：规范约束

### 必须满足
| 约束 | 说明 |
|------|------|
|      |      |

### 不允许
| 约束 | 说明 |
|------|------|
|      |      |

### Gate 定义
```
CP3 Verify 通过当且仅当：
  条件 A === true AND
  条件 B === true AND
  ...
```
```

**Gate 定义规则**：
- 用 `AND` 连接所有必须满足的条件
- 每个条件都是可验证的（能跑命令/能手动检查）
- 没有"或者"——只有全部通过 or 全部不通过

## 填充示例（以 refactor 为例）

### 第一节：操作过程

```markdown
### CP0: 初始化
- 输入：需要重构的代码文件或模块
- 输出：重构范围定义

### CP1: 规划
- 分析代码依赖关系
- 确定重构边界
- 制定重构步骤
- Human approve 后进入 CP2

### CP2: 执行
1. 提取公共函数
2. 消除重复代码
3. 简化复杂逻辑
4. 每步 commit

### CP3: 验证
- lint + typecheck + test 全部通过
- 重构前后行为一致（对比测试结果）
- 性能不下降

### CP4: 完成
- 更新 state.json
- 记录新的代码模式到 patterns
```

### 第二节：决策启发

```markdown
### 遇到循环依赖
```
A 引用 B，B 引用 A →
  ├─ 同一模块内 → 合并或提取到新模块
  ├─ 跨模块 → 引入中间层解耦
  └─ 无法拆分 → 停下来，报告人类
```

### 重构影响范围超出预期
```
重构过程中发现影响其他模块 →
  ├─ 评估影响 < 2 个模块 → 一起改
  └─ 影响 > 2 个模块 → 停止，回到 CP1 重新规划
```
```

### 第三节：规范约束

```markdown
### 必须满足
| 约束 | 说明 |
|------|------|
| lint 通过 | exit code = 0 |
| typecheck 通过 | exit code = 0 |
| 测试通过 | 全量测试 pass |
| 行为不变 | 重构前后测试结果一致 |

### 不允许
| 约束 | 说明 |
|------|------|
| 改变公共 API 行为 | 重构是内部优化，不改对外契约 |
| 降低测试覆盖率 | 可以不变，但不能降 |
| 引入新的 lint 警告 | 清理旧警告，不添新警告 |

### Gate 定义
```
CP3 Verify 通过当且仅当：
  lint === pass AND
  typecheck === pass AND
  test === pass AND
  behavior_unchanged === true
```
```

## 快速检查清单

创建新 Skill 前，问自己：

- [ ] 操作过程是否覆盖了从触发到完成的完整流程？
- [ ] 决策启发是否覆盖了所有可能的分支？
- [ ] Gate 定义中的每个条件是否可验证？
- [ ] 是否有"不允许"条款防止常见错误？
- [ ] Skill 是否依赖其他文件（constraints、autonomy 等）？是否引用了它们？
