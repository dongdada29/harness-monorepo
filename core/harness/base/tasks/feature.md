# Feature Template - 功能开发模板

> 适用于所有项目  
> **版本**: 2.0.0（新增决策启发 + 规范约束）

---

## 结构总览

```
完整 Skill = 操作过程 + 决策启发 + 规范约束

操作过程 → 第一节（做什么）
决策启发 → 第二节（怎么做分支判断）
规范约束 → 第三节（做到什么程度算合格）
```

### CP1: 规划

- [ ] 确认功能范围清晰（包含/不包含明确）
- [ ] 技术方案至少覆盖主要文件
- [ ] 验收标准可量化（不接受"体验更好"这种模糊表述）
- [ ] 实现计划拆到可执行的子任务
- [ ] **Human approve** 后进入 CP2

### CP2: 执行

- [ ] 按实现计划逐步执行
- [ ] 每完成一个子任务，更新 state.json 的 checkpoint
- [ ] 遵守日志规范（参考 `logging.md`）
- [ ] 每次 commit 保持原子性（一个子任务一个 commit 最佳）

### CP3: 验证

- [ ] lint 通过
- [ ] typecheck 通过（如果有）
- [ ] 测试通过（单元测试 + 集成测试）
- [ ] 构建通过
- [ ] 验收标准逐条核对

### CP4: 完成

- [ ] 更新 `state.json` 的 `tasksCompleted` +1
- [ ] 如果是新模式（如新的 API 用法），追加到 `patterns` 字段

## 第二节：决策启发 (Decision Heuristics)

> 遇到以下情况时，按照对应分支处理

### 需求模糊

```
需求描述不清晰 →
  ├─ 查 Semantic Memory（项目规范、技术约束）
  ├─ 有明确规范 → 按规范执行
  └─ 无规范或规范矛盾 → 在 CP1 停下来，向人类确认
  注意：不要猜。猜对了也是坏习惯。
```

### 技术方案选型

```
遇到多种实现方案 →
  ├─ 优先复用现有模式（查 Semantic Memory 的 patterns）
  ├─ 有现成模式 → 用现成的
  └─ 没有现成模式 → 选侵入性最小的方案，说明理由
```

### 涉及 API 变更

```
功能涉及现有 API 修改 →
  ├─ 优先后兼容（加参数、加 overload，不改签名）
  ├─ 无法后兼容 → 在 CP1 停下来，明确说明破坏性变更
  └─ 必须破坏性变更 → 同步更新所有调用方（记录在 pendingTasks）
```

### 自测覆盖不足

```
测试覆盖率 < 80% →
  ├─ 先看是否是测试配置问题（istanbul/vitest 配置）
  ├─ 确认是真实覆盖不足 → 补测试到 80%+ 再进入 CP4
  └─ 无法达到 80%（如纯 UI 组件）→ 说明理由，人工 approve
```

### 实现过程中发现范围蔓延

```
实现过程中发现需求漏掉了功能点 →
  ├─ 同级功能点 → 记录到 pendingTasks，不阻塞当前任务
  ├─ 核心功能点影响原方案 → 停止 CP2，回到 CP1 重新规划
  └─ 小细节且明确 → 评估 < 30 分钟则直接做，否则记录
```

### 阻塞（30 分钟无进展）

```
阻塞处理 →
  ├─ 查 Memory（CP2 EXEC 检索）→ 有相似案例则复用
  ├─ 无相似案例 → 升级到 CP3，人工介入
  └─ 连续 3 次同类阻塞 → 触发 autonomy.md L3 人工介入
```

## 第三节：规范约束 (Normative Constraints)

> **Gate 定义**：以下条件必须全部满足，CP3 才能标记为 completed

### 必须满足

| 约束 | 说明 |
|------|------|
| **lint 通过** | `npm run lint` 或项目对应 lint 命令，exit code = 0 |
| **typecheck 通过** | `tsc --noEmit` 或项目对应命令，exit code = 0 |
| **测试通过** | 单元测试 + 集成测试全部 pass |
| **构建通过** | `npm run build` 或等效命令，exit code = 0 |
| **验收标准 100% 满足** | 每条标准必须有对应的测试或手动验证记录 |
| **测试覆盖率 ≥ 80%** | 新增代码的覆盖率，不接受降低现有覆盖率 |
| **每次 checkpoint 更新 state.json** | 每个子任务完成后记录一次 |

### 不允许

| 约束 | 说明 |
|------|------|
| **不允许跳过 lint/typecheck** | 即使很自信，也必须跑 |
| **不允许强制提交** | `git push -f` 禁止 |
| **不允许降低测试覆盖率** | 如果现有覆盖率 < 80%，这是项目问题，不允许继续降低 |
| **不允许破坏性变更不通知** | 必须显式说明破坏了什么 |

### 合格标准

```
CP3 Verify 通过当且仅当：
  lint === pass AND
  typecheck === pass AND
  test === pass AND
  build === pass AND
  acceptance_criteria === 100% AND
  coverage >= 80%
```
